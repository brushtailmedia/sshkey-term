### Added
- **V3 `room_added_to` live room-add notification (client) (2026-05-22).** When an operator adds the local user to a room while they're connected, the terminal now adds the room to the sidebar + member cache immediately — no reconnect needed (companion to the server-side V3 work in the sshkey-chat repo). The client-layer handler hydrates the keyed room-member cache (`setRoomMembers`) and persists the room via the V8 `UpsertRoomWithMembers` helper (member-count + IDs in one write) plus `MarkRoomRejoined` (un-archiving a previously-left room), logging store/unmarshal failures rather than silently dropping them. The TUI handler inserts the room via a new `SidebarModel.AddRoom` (by-ID, clearing any stale left/retired grey) and surfaces a status-bar toast + desktop notification **without stealing focus** from the current conversation; if the re-added room is the one being viewed, it refreshes the messages pane's read-only state in place so compose reactivates. The actor label special-cases the CLI's `os:<uid>` form (rendered "server admin") rather than the raw `os:501` — a difference from `group_added_to`, whose adds always come from a real user. New `protocol.RoomAddedTo` struct. Tests cover the client handler (cache + store + `MarkRoomRejoined` + malformed-input safety) and the TUI handler (sidebar add without focus steal, os: vs real-user label, viewed-re-add reactivation). Follow-up audit required no term runtime change; server-side V3 polish tightened CLI partial-failure wording and higher-level room-add tests in sshkey-chat.

- **V8 room-member cache storage foundation (2026-05-22).** The protocol mirror now treats `room_list.members` as full member user IDs, and the term decoder's server-to-client frame cap is raised to 8 MiB so larger room lists do not fail the handshake. Member ID normalization now trims and drops blank IDs, de-dupes while keeping first occurrence, preserves server order, and returns defensive copies.

- **V8 room-member cache wiring + read-only-room policy — resolves the member-panel/info-panel refetch storm (2026-05-22).** Builds on the storage foundation above to complete the client side of V8. The client hydrates the in-memory room-member cache (`roomMemberCache`) from the local DB on startup and maintains it live: `room_list` snapshots overwrite it (and persist via `UpsertRoomWithMembers`), `room_event` join/leave deltas mutate it in place — the same snapshot-persists / delta-in-memory model groups already use. The info panel and the right-side member panel now render room members straight from this cache. Previously each panel-open, room-switch, and member-panel toggle issued a `room_members` request — the "refetch storm" — and a member panel could render blank until a manual refresh landed. Automatic UI paths no longer fetch: the **only** remaining production `RequestRoomMembers` call site is the explicit `r` refresh, locked by a static guard test (`internal/tui/request_room_members_guard_test.go`) modeled on the path-drift guard. Retired/left rooms are now treated as read-only in the client: the info and member panels drop the member-list UI and render a static notice instead of rows (distinguishing a genuinely read-only room from an active-room cache miss, which shows "(members unavailable — press r to refresh)" so an empty cache never reads as a zero-member room), and the input accepts only the allow-list `/delete`, `/search`, `/help`, `/?` — normalized (trim + lowercase verb, trailing args ignored) and enforced at the dispatch layer (`handleSlashCommand`), not merely the keypress layer, with a typing-time completion hint built directly from the allow-list. ### Fixed
- **Room join no longer renders "os:&lt;uid&gt; added X to the room" (2026-05-22).** A follow-up to the V3 room-add work: V3 routed default-room / approve auto-joins through the same `pending_add_to_room` queue as manual adds, so those paths began recording persisted `room_event{join}` audit rows (previously they wrote `room_members` directly and recorded nothing). For rooms — which are operator-managed, so the actor is always an opaque `os:<uid>` — that surfaced a low-value "os:0 added X to the room" line. The join event is also stamped at *queue-processing* time (~5s after the add), so it lands just after the newly-added user's connect-time sync watermark; the room's `sync_batch` catch-up (`ts >= last_synced`) therefore re-sent it on the next reconnect, and because room events render only for the *active* room it appeared inconsistently — e.g. in `general` but not `support` when both are default rooms (whichever was active during the reconnect). The TUI no longer renders `room_event{join}` as an inline system message: room membership is conveyed by the member panel + the room cache, and the newly-added user still gets the `room_added_to` toast. Room **leave / topic / rename / retire** still render, and **group** join messages are unchanged (peer-admin adds are meaningful). The server still queues the join (so epoch rotation and the existing-member broadcast for manual `add-to-room` are unaffected) — only the client-side render is dropped. Regression tests in `internal/tui/room_join_render_v3_test.go` (join not rendered; leave still rendered).

- **V8 follow-ups: open panels now reflect live membership, and re-added rooms recover in-session (2026-05-22).** Two gaps found auditing the V8 cache work, both verified against the implemented code before fixing. **(1) Open info/member panels were only partially live.** They already re-derived display fields (presence / status / verified / admin) on every render, but the member-ID *set* was frozen at panel-open time — a member who joined or left while the panel was open did not appear/disappear until the panel was reopened (info) or the context switched (member panel). An App-level refresh bridge now pushes the current member set + live state into the open panel before its `Update` and `View`: the new `InfoPanelModel.SetLiveMemberIDs` (which replaces the display-only `derivedMembers`/`derivedAdmin` helpers and the panel's outward `*client.Client` reference) and `MemberPanelModel.RefreshPreservingSelection` both rebuild rows from the live cache while preserving the selected *user* (not the bare cursor index) and re-clamping the cursor. The info panel also re-derives the local group-admin gate (so a live self-promote/demote updates the `a/r/p/x` footer actions, not just per-row badges), transitions to read-only if the room is retired/left while open, and fills in a DM's other party once the cached pair hydrates. The member-panel refresh is wired before its `Update`, `View`, **and** the mouse-hit path (so a click can't select a stale row when the rendered set just changed) and re-feeds the `@`-completion sources on the persistent (non-render) paths. Supporting changes: `client.GroupMembers` now returns a defensive copy (matching `RoomMembers`, since the bridge reads it every cycle), and `MemberPanelModel.SetRoomMembers` clamps the cursor so a shorter explicit-refresh response can't leave it past the end. The View-path refresh deliberately only freshens the rendered copy (`viewBody` is a value receiver) — persistent action state is owned by the `Update` and mouse paths. **(2) A re-added room stayed visually dead within the session.** When an admin re-added the user and a fresh `room_list` cleared the store's `left_at`, the sidebar row stayed greyed and — if the user was viewing it — the compose pane stayed read-only until a context switch, because the sidebar marker loop only ever *added* the greyed flag (no `else` to clear it) and the handler never refreshed the messages pane. The `room_list` handler now clears stale sidebar grey via `MarkRoomRejoined` in the marker loop's `else`, and calls `syncMessagesLeftState()` after reconciliation to reactivate the compose pane in place. The pre-existing dead re-add loop (the client-layer handler clears `left_at` first, so its `IsRoomLeft` guard never fired) was removed and replaced with a comment redirecting future edits to the marker loop. New tests: `internal/tui/infopanel_derive_v8_test.go` (rewritten against `SetLiveMemberIDs` — display + membership freshness, selected-user preservation, cursor clamp, group-admin source + local gate, DM hydration, live read-only transition, no-op cases), `internal/tui/member_panel_live_v8_test.go` (preserving refresh, clamp, read-only/cache-miss notice, `SetRoomMembers` clamp, App-level persistent refresh + completion), `internal/tui/room_readd_reconcile_v8_test.go` (Gap A sidebar, Gap B compose pane, still-left negative), and a `GroupMembers` defensive-copy test in the client suite. Two pre-existing mouse-click tests were updated to seed the member panel via the cache (the production path) now that the mouse handler refreshes from it. Full `internal/tui` + `internal/client` suites green, including under `-race`. Design + audit notes in the gitignored `fix-v8-panel-liveness-and-rejoin-followups.md`.

- **Two post-stale-event-guard cleanups (2026-05-21).** Follow-ups from the gen-token audit. **(1) `switchServerByIndex` now validates the target host before closing the current client.** Pre-fix the function disconnected the active session first, then ran `config.ValidateHost(srv.Host)`, so a hand-edited invalid Host in `config.toml` (path separator, traversal segment, control byte — the same shapes the wizard rejects) would kick the user off the working server *and then* refuse the switch, leaving them with no connection at all. Reordered: validate target, refuse with a status-bar error on failure (current session untouched), close + switch only when the target is known good. Two regression tests in `internal/tui/switch_validate_test.go`. **(2) Removed dead `ConnectedMsg` / `ReconnectStatusMsg` types.** Both were declared in `internal/tui/app.go` but had **zero production emitters** (`grep` came up empty across the repo). `ReconnectStatusMsg` even had a full three-case handler (`reconnecting`/`connected`/`failed`) that was therefore unreachable. Removed the type declarations, the unreachable handler, and refreshed two stale comments in `app.go` and `connectfailed_app_test.go` that still referenced `ConnectedMsg`. Why this matters beyond cleanup: a future reuse of either type would land on the gen-token contract footgun — a new emitter would have no gen-stamp, the handler would have no gen-check, and a stale message after a server switch would silently mutate UI state. Removing them now closes that re-entry point. The actual reconnect/connect flow uses `connectedWithClient` and `ErrMsg`, both already gen-stamped.

- **Stale events from a superseded connection could mutate UI state for the wrong server (2026-05-21).** During a server switch (or reconnect race, or retry from the connect-failed overlay), a slow background event from the prior `connect()` could still arrive after the user had moved on — there was no event-origin identity, so a stale `connectedWithClient` would overwrite the active `a.client`, a stale `ServerMsg` would be decoded and applied against the new server's context, a stale `ErrMsg` would show the connect-failed overlay for a connection the user had already abandoned, a stale `passphraseNeededMsg` would open the dialog for the prior server's key, and stale `RoomUpdatedEvent` / upload / download / save / `AttachmentReadyEvent` / `KeyChangeEvent` / reconnect-timer events would surface status / preview / modal mutations for a session that no longer existed. Fixed with a connection-generation guard: a new `App.connGen uint64` (seeded to 1 in `New()`, never zero so missing-stamp bugs surface immediately) bumps via `nextConnGen()` on every fresh connect attempt — server switch, reconnect timer, retry-from-overlay, passphrase retry, refresh reconnect. All twelve connection-scoped Msg structs (`ServerMsg`, `ErrMsg`, `connectedWithClient`, `passphraseNeededMsg`, `PassphraseResultMsg` + `keyPath`, `reconnectAttemptMsg`, `KeyChangeEvent`, `AttachmentReadyEvent`, `RoomUpdatedEvent`, `UploadResultEvent`, `DownloadResultEvent`, `SaveResultEvent`) carry the producing `gen`; `previewRenderReadyMsg` deliberately stays unstamped because it already validates against a request-specific `previewRenderKey` that isn't connection-scoped. The signature of `connect()` is now `connect(gen uint64)` — the compiler enforces explicit gen-passing, so no future call site can accidentally reach a no-arg `connect()`; a new `startConnect()` helper bumps + calls in one step for every non-initial attempt. `waitForMsg` accepts `gen` and stamps both the `errCh` wrap and the synthetic `done`-channel `ErrMsg{disconnected}`. Each gen-scoped `updateInner` case checks `msg.gen != a.connGen` as its first meaningful operation and drops with no UI mutation, status update, modal change, or wait re-arm — except `connectedWithClient`, which closes `msg.client` before dropping so a slow-winning stale connect doesn't leak its readLoop / SSH transport / store handle. `PassphraseModel.Show` now binds the gen + keyPath onto the model so emitted `PassphraseResultMsg`s carry them through the dialog cycle; the cache key is now `msg.keyPath` (not `a.cfg.KeyPath`), so a switch-then-submit doesn't cache server A's passphrase under server B's path. `RefreshRequestMsg{Kind:"reconnect"}` now closes the old client AND immediately calls `startConnect()` instead of relying on the old `done` event (which is now stale and dropped). `switchServerByIndex` dismisses the passphrase modal on switch. Upload/download/save goroutines capture `a.connGen` at launch and stamp their result events; `open_attachment` carries a caveat — the OS-level `OpenFile` still fires inside the goroutine even if the user switched servers in the brief download window (strict cancellation deferred per the doc; status text is still gen-dropped). 24 new regression tests in `internal/tui/gen_token_test.go` cover the full stale-drop matrix (stale `connectedWithClient` closes the client + doesn't overwrite `a.client`, stale `ServerMsg` is a no-op, stale `ErrMsg` shows no overlay / schedules no reconnect, stale passphrase req/result paths don't open dialog / cache / start connect / quit on Cancelled, stale reconnect timer is dropped, stale upload/download/save/attachment/room-updated don't mutate status or preview, current-gen paths still install client / cache under msg.keyPath / process normally, `nextConnGen` bumps monotonically, `startConnect` bumps before capturing — locking the bump-order invariant, `waitForMsg` stamps `done` and `errCh`-wrapped synthetic events with gen). Per-server DB isolation is unchanged (per-client `c.store` already binds correctly; this fix is purely UI/ownership correctness, not the original DB-contamination framing the design doc started from). Design + acceptance criteria + implementation-risk catalogue in the gitignored `fix-cross-server-db-isolation.md`.

### Fixed
- **Encrypted-key first-launch deadlock (2026-05-14).** Pre-fix, users who followed the wizard's "Passphrase (recommended)" prompt and generated an encrypted Ed25519 key hit a deadlock on every subsequent launch: the TUI displayed `Connecting...` indefinitely, no SSH dial appeared in `client.log`, and the passphrase dialog never showed. The only escape was Ctrl+C → quit → delete config → re-run the wizard with an empty passphrase. Root cause: `internal/client/keys.go`'s `loadSSHKey` was changed at some point to consume the `OnPassphrase` callback (calling it and waiting for the bytes), but `internal/tui/app.go`'s `connect()` still expected `client.Connect()` to RETURN a `"passphrase"`-substring error — the dialog trigger (`passphraseNeededMsg{}`) was only dispatched in the `if err := c.Connect(); err != nil` branch. Two halves of the design that don't compose: `loadSSHKey` blocks inside `OnPassphrase`'s `<-passphraseCh` receive waiting for a passphrase that never arrives, while the dialog that would prompt the user is only triggered after `Connect()` returns, which it never does. Classic deadlock. Goroutine parked indefinitely; TUI rendered `app.View()`'s `Connecting...` fallback (`a.connected` false, `a.err` nil because Connect never errored — it's just stuck).

  Fix: pre-flight an encryption check in `App.connect()` before calling `c.Connect()`. New exported helper `client.KeyNeedsPassphrase(path string) (bool, error)` in `internal/client/keys.go` peeks at the PEM file via `ssh.ParsePrivateKey` and reports whether it would return `*ssh.PassphraseMissingError`. When the cache is empty AND the pre-flight reports `true`, `connect()` returns `passphraseNeededMsg{}` immediately — before any `Connect()` call, before any goroutine parking. The existing `PassphraseResultMsg` handler caches the typed passphrase and re-runs `connect()`; on the retry, the cache is populated, the pre-flight is skipped, `OnPassphrase` returns the cached bytes synchronously inside `loadSSHKey`, and `Connect()` proceeds normally. Net cost on the cold encrypted path: one extra `os.ReadFile` + `ssh.ParsePrivateKey` (microseconds). Warm reconnects with cached passphrase skip the pre-flight entirely (zero added cost).

  Secondary fix: drain-then-send in the `PassphraseResultMsg` handler at `app.go:2986`. `passphraseCh` is a buffered channel (cap 1). The handler's `a.passphraseCh <- msg.Passphrase` send normally completes because the buffer has room — but if a prior `OnPassphrase` cycle left a stale byte slice in the buffer (e.g., a future reconnect path that clears the cache and re-prompts), the send would BLOCK and hang Update — a second deadlock. The fix drains any stale value via `select { case <-a.passphraseCh: default: }` before sending. Belt-and-suspenders: the cache also serves as the source of truth, so the channel send is now mostly redundant on the retry path; the drain just guarantees the buffer state stays clean for future direct-channel consumers.

  Defensive fallback retained at `app.go:584-590`: the existing `strings.Contains(errStr, "passphrase")` branch becomes effectively unreachable under the pre-flight design (the pre-flight catches the case first; `Connect()` either succeeds via cached bytes or fails with a non-passphrase error). Kept as defence-in-depth so a future regression in pre-flight detection wouldn't silently re-introduce the original hang.

  **Companion view-layer fix:** `App.viewBody()` had a parallel bug — the not-connected fallback (`return "\n  Connecting...\n"`) returned BEFORE checking `a.passphrase.IsVisible()` (the passphrase-rendering branch at the bottom of the function only ran inside the connected path). So even after the pre-flight correctly dispatched `passphraseNeededMsg` and the handler correctly called `a.passphrase.Show("")`, the dialog stayed invisible-to-the-user — visible at the model level but never rendered, because the view bailed out at "Connecting..." first. The `connectFailed.IsVisible()` check at the top of `viewBody` had been correctly hoisted above the not-connected bail-out; the parallel passphrase check was missed. Fix: hoist the `passphrase.IsVisible()` check to the same position. Regression test `TestApp_PassphraseDialogRendersWhenNotConnected` builds an App with `connected=false` + passphrase shown, calls `viewBody()` directly, and asserts the rendered output contains the dialog's distinctive rounded-border glyphs (`╭`/`╰`) rather than the "Connecting..." string. This was the actual user-visible symptom — terminal sits at "Connecting..." with the dialog rendered behind it but never displayed.

  **Why the bug went undetected:** the wizard allows empty passphrase (it's labeled "(recommended)" but not enforced), and manual single-server testing happened to use unencrypted keys throughout. For unencrypted keys, `ssh.ParsePrivateKey` succeeds without entering the `*ssh.PassphraseMissingError` branch, so `OnPassphrase` was never called, so the deadlock never triggered. `internal/client/keys_test.go` tests `loadSSHKey` with a synchronous passphrase function (no channel), which exercised the keys-package contract in isolation but didn't surface the channel-based callback's interaction with `App.connect()`. The bug only manifests when the App's channel-based callback meets `loadSSHKey`'s callback-consumer — an interaction not previously tested at any layer.

  **Tests** in new `internal/tui/passphrase_flow_test.go`: `TestApp_EncryptedKeyDispatchesPassphraseNeeded` generates an encrypted key fixture and asserts `connect()` returns `passphraseNeededMsg` within 200ms (a 200ms budget on a sub-millisecond-fast pre-flight; failure means deadlock regression). `TestApp_UnencryptedKeyProceedsToConnect` asserts an unencrypted key does NOT trigger the prompt (fall-through to Connect). `TestApp_EncryptedKeyWithCachedPassphraseSkipsPrompt` pre-populates the cache and asserts the warm path skips the dialog. All three tests use timeout guards (split-budget design: 200ms for the positive case, 50ms for the negative-case fall-through) so a regressed deadlock fails fast rather than hanging the test process forever.

### Security
- **Path traversal via sender-supplied attachment filename.** Pre-fix: `att.Name = "../../etc/cron.d/evil"` combined with `filepath.Join(home, "Downloads", att.Name)` in the old save handler would let a hostile peer steer a save outside the chosen destination directory whenever the user pressed `s`. - **Save-as download goroutine never scheduled (non-cached attachments).** Pre-fix `case "save_attachment"` at `internal/tui/app.go:1818` used `cmds = append(cmds, downloadFn)` — but the enclosing `case MessageAction:` block exits with `return a, nil`, which discards the `cmds` accumulator entirely.
